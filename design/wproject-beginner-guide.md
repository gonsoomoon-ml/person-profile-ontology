## 0. 한 줄 요약

DoorDash 같은 **음식배달 회사**가 "이 고객은 어떤 사람인가"를 파악(**User Profiling**)해서 **더 잘 맞는 추천**을 주려고 합니다. 우리는 이걸 **관계 지도(그래프=온톨로지)** 위에서 만드는 방법을 연구했고, 먼저 **작게 노트북으로 검증**한 뒤 필요하면 **대규모로 확장**하는 계획을 세웠습니다.

---

## 1. 무엇을 하려는 건가요? (문제)

- 고객이 주문하면 데이터가 쌓입니다: **누가 / 어떤 식당에서 / 어떤 메뉴를 / 언제**.
- 이 데이터로 **고객 프로필**을 만들면 → 맞춤 추천, 안전한(알레르기·식이) 추천, 신규 고객 첫 추천이 가능해집니다. → **고객 경험 ↑, 주문 ↑.**
- 우리가 잡은 핵심 인사이트: **"말한 것(stated) vs 실제 한 것(revealed)"의 차이.**
  - 예) *"난 이탈리안 좋아해"* 라고 가입 때 골랐지만 → 실제론 매번 멕시칸 주문 → **진짜 취향은 멕시칸.**
  - 교훈: **말보다 행동이 진실에 가깝다.** 이 차이를 잡는 게 프로파일링의 핵심 가치.

---

## 2. 그래프(온톨로지)란? — "관계 지도"

**비유: 지하철 노선도.** 역(점) = 사람·식당·요리, 노선(선) = 관계.

- **노드(node)** = 점. 예: `User`(사용자), `Restaurant`(식당), `Cuisine`(요리종류), `Dish`(메뉴), `Region`(지역), `Order`(주문).
- **엣지(edge)** = 선 = 관계. 예: `PLACED`(주문했다), `AT`(그 식당에서), `SERVES`(그 요리를 낸다).
- **온톨로지(ontology)** = 이 관계 지도에 **의미(타입)** 를 붙인 것. 그냥 점·선이 아니라 *"사람이 주문을 했고, 그 주문은 어떤 식당에서 처리됐다"* 처럼 **규칙이 있는 지도**.

### 우리 데모 그래프 구조 (실제로 도는 것 — `neptune-demo`)

```
  LIKES_CUISINE  (말한 선호 · stated) ───────────────────────────────────────┐
  ┌────────┐                                                                ▼
  │  USER  │   PLACED    ┌────────┐    AT     ┌────────────┐  SERVES   ┌──────────┐
  │ 사용자 │ ──(주문)──▶ │ ORDER  │ ────────▶ │ RESTAURANT │ ────────▶ │ CUISINE  │ ★허브
  │--------│            │ 주문   │           │   식당     │           │ 요리종류 │
  │ name   │            │ total  │           │ priceTier  │           │ name     │
  │ diet   │            └───┬────┘           │ rating     │           └────▲─────┘
  │(속성)  │                │ CONTAINS       └─────┬──────┘                │
  └───┬────┘                ▼                      │ LOCATED_IN      OF_CUISINE
      │                ┌────────┐                  │                       │
      │ LIVES_IN       │  DISH  │ ─────────────────┼───────────────────────┘
      │ (사는 곳)      │ 메뉴   │   OF_CUISINE      │
      ▼                └────────┘                   ▼
  ┌──────────────────────────────────────────────────────┐
  │                    REGION (지역)  ★허브                 │
  └──────────────────────────────────────────────────────┘

  ★허브 = 여러 선이 모이는 점. (분석 결과: Cuisine이 가장 중심 — 모든 식당·메뉴가 요리로 모임)
  규모: 사용자 8명 · 주문 27건 (작지만 의미 있는 쿼리가 도는 미니 데모)
```

### 그래프의 역할 — "왜 그래프인가?"

그래프는 **데이터 저장고가 아니라**, 아래 **3가지를 특히 잘하는 도구**입니다:

1. **관계 따라가기(탐색)** — "이 사람과 비슷한 사람", "이걸 좋아한 사람이 좋아한 다른 것" (= **look-alike** / 협업 추천).
2. **설명(왜 추천?)** — *"매운 한식을 자주 시켜서 이 식당을 추천"* 처럼 **선을 따라가며 이유**를 보여줌.
3. **안전(제약)** — *"비건이니까 고기 메뉴는 제외"* — 연결을 통해 위험한 메뉴를 걸러냄.

> 💡 거꾸로, 이 3가지가 필요 없고 **점수만** 필요하면 그래프는 **과한 도구**입니다. 그땐 더 단순·저렴한 방법으로 충분 (→ 6장 우선순위).

## 3. 그래프 하나로는 부족 — "3층 구조"

수천만 명 규모에선 **모든 데이터를 그래프에 넣으면 안 됩니다** (느려지고 비쌈). 그래서 역할을 셋으로 나눕니다.

**비유로 이해하기:**

| 층 | 비유 | 하는 일 |
|---|---|---|
| **그래프(Graph)** | **관계 지도** | 누가 무엇과 연결됐나 (탐색·설명·안전) |
| **피처 스토어(Feature Store)** | **즉석 메모 카드함** | 자주 쓰는 계산 결과(예: 이 고객 RFM 점수, 최애 요리)를 미리 적어두고 **아주 빠르게(밀리초) 꺼냄** |
| **OLAP(데이터 웨어하우스)** | **대형 창고 + 계산실** | 모든 영수증(전체 주문 기록) 보관 + **대량 집계**(밤에 한꺼번에) |

**무엇을 어디에 두나:**

| 데이터 | 어디에 | 왜 |
|---|---|---|
| 관계 (사람–식당–요리) | **그래프** | 탐색·설명·안전 |
| 자주 읽는 결과값 (RFM, 최애 요리, 점수) | **피처 스토어** | 빠른 응답 |
| 전체 주문 기록 · 무거운 집계 | **OLAP** | 대량 처리 |
| 끝없이 쏟아지는 **새 주문** | 그래프에 직접 ❌ → **스트림으로** | 직접 쓰면 그래프가 막힘 |

> 🔑 **한 줄 규칙: 관계는 그래프, 속도는 피처 스토어, 양은 OLAP.**

## 4. 언제 계산하나? — 실시간 vs 배치

- **배치(batch) = 매일 밤 미리 숙제 해두기.** 예: RFM·취향·"말 vs 행동" 차이를 밤에 계산해 피처 스토어에 저장. → **대부분이 여기.**
- **실시간(real-time) = 손님이 올 때 그 자리에서.** **꼭 즉시 해야 하는 것만:**
  1. **안전 필터** — 방금 "비건"으로 바꿨으면 **즉시** 반영 (하루 못 기다림).
  2. **최종 추천 필터·순위** — 방금 주문한 곳 빼기, 위치·시간 반영.
  3. **이유 설명** — 이 추천의 경로(why).
  4. **신규 고객** — 기록이 없으니 그 자리에서 대체 추천(사는 지역·말한 취향 기반).

> 🔑 정리: **"거의 다 배치, 안전·즉시성만 실시간."**

## 5. 진짜 대규모(3천만 명 / 일 1천만 활성)에서 가능한가요? (feasibility)

**가능합니다 — 우리보다 훨씬 큰 회사들이 이미 매일 이렇게 돌리고 있기 때문입니다.**

- **Pinterest (Pixie)**: 30억 점 / 170억 선 그래프를 **한 대의 컴퓨터 메모리**에 올리고 **매일 새로 적재**. → 우리 규모(사용자 3천만)는 이보다 **훨씬 작음**.
- **Uber**: 고객 특징을 **하루 한두 번** 미리 계산해 저장하고 빠르게 서빙.

**단, 이걸 지켜야 가능합니다 (조건):**
- 전체 주문 기록을 그래프에 **다 넣지 않기** (창고=OLAP에).
- 새 주문을 그래프에 **1건씩 즉시 쓰지 않기** (모아서/스트림으로).
- 매일 "**전체를 새로 짓기**"가 아니라 "**계산값만 다시 계산**".

> ⚠️ 주의: 위 수치는 "남들이 더 큰 규모로 한다"는 **가능성의 증거**일 뿐, 우리 서비스의 성능 보장은 아닙니다. → 그래서 **작게 시작해 실제로 측정**합니다.

## 6. 비용 / 우선순위 — "싼 것부터"

컴퓨팅 비용이 큽니다. 그런데 **돈 되는 가치는 대부분 싼 부분**에 있습니다.

- 🟢 **먼저 (싸고 효과 큼):** 배치 프로필(RFM·취향) + 간단 추천 + 안전 필터 + 신규고객 대체. → **그래프/실시간/딥러닝 없이도 약 80% 가치.**
- 🟡 **다음 (가치 증명되면):** look-alike, 그래프 DB(설명·안전이 정말 중요할 때만), 안전을 위한 실시간 슬라이스.
- 🔴 **나중 / 보류 (비싸고 효과 불확실):** 딥러닝 임베딩(two-tower), GNN, "모든 걸 LLM으로".

**비용 절약 핵심:**
- **활성 고객(1천만)만** 계산 (전체 3천만 매일 X). ← 가장 큰 절약.
- **가치에 따라 주기 조절** (마케팅 분석용은 매일 말고 **주간**).
- **안 쓸 값은 계산 안 함** (추천/결정을 안 바꾸면 계산 불필요).

> 🔑 **큰 기계를 처음부터 짓지 마세요.** 노트북으로 효과를 먼저 증명하고, **돈을 버는 게 증명된 비싼 층만** 나중에 추가합니다. (그래프·실시간·임베딩은 *기본값이 아니라 벌어서 얻는 것*.)

## 8. 용어 사전 (초보자용)

| 용어 | 쉬운 뜻 |
|---|---|
| **그래프(graph)** | 점(노드)과 선(엣지)으로 관계를 그린 지도 |
| **노드 / 엣지** | 점(사물) / 선(관계) |
| **온톨로지(ontology)** | 의미(타입)가 붙은 관계 지도 |
| **피처 스토어(feature store)** | 미리 계산한 값을 빠르게 꺼내는 "메모 카드함" |
| **OLAP / 데이터 웨어하우스** | 대량 데이터를 보관·집계하는 "창고+계산실" |
| **배치(batch)** | 미리 한꺼번에 계산 (밤에 숙제) |
| **실시간(real-time)** | 요청이 올 때 그 자리에서 계산 |
| **CDC** | 데이터가 바뀔 때마다 자동으로 알려주는 변경 알림 |
| **RFM** | 최근성·빈도·금액 — 고객 가치 점수 |
| **콜드스타트(cold-start)** | 기록 없는 신규 사용자에게 추천하는 문제 |
| **supernode** | 연결이 너무 많은 점(인기 식당) — 따로 정리 필요 |
| **look-alike** | 취향이 닮은 사용자(유사 타깃) |
| **stated vs revealed** | 말한 선호 vs 실제 행동으로 드러난 선호 |
